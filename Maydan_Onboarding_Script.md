# Maydan Onboarding Script

## Goal

Use this script for short live walkthroughs with Admin, submitters, approvers, and leadership during rollout.

Target length:
- Admin walkthrough: 15 minutes
- Submitter walkthrough: 10 minutes
- Approver walkthrough: 10 minutes

## Admin Walkthrough

### Intro

Today we are reviewing Maydan as the internal system for staff event requests, approvals, notifications, and calendar coordination.

By the end of this walkthrough, you should be able to manage users, entities, facilities, and track event workflow status.

### Steps

1. Log in at `https://maydan-six.vercel.app`
2. Open `Dashboard`
3. Review the summary cards, recent activity, and approval visibility
4. Open `Admin > Users`
5. Review how to add, edit, and deactivate a user
6. Open `Admin > Entities`
7. Review how entity heads are assigned
8. Open `Admin > Facilities`
9. Review how facilities are added, edited, and deactivated
10. Open `Admin > Audit`
11. Review how to inspect event history and filter by status/entity/date

### Admin Sign-Off Questions

- Can you find the correct user, entity, and facility records?
- Do the current entity heads look correct?
- Does the audit view show enough workflow history?
- Are there any staff accounts that still need correction before full launch?

## Submitter Walkthrough

### Intro

Maydan is the place to submit staff events and track where they are in the approval chain.

### Steps

1. Log in
2. Open `Events`
3. Review existing drafts, pending events, and approved events
4. Click `Submit New Event`
5. Fill out the event form
6. Point out:
   - facility selection
   - audience selection
   - grade level
   - marketing toggle
7. Submit the event
8. Open the new event detail page
9. Review:
   - status badge
   - approval chain progress
   - revision handling if an event is rejected

### Submitter Questions

- Was the form easy to understand?
- Were any fields unclear?
- Did the event detail page make the next step obvious?

## Approver Walkthrough

### Intro

Maydan routes events to approvers automatically based on the submitter's entity.

### Steps

1. Log in
2. Open `Approvals`
3. Open an event waiting on your step
4. Review the event details and current chain
5. Review the three actions:
   - `Approve`
   - `Reject`
   - `Suggest Alternative`
6. Demonstrate how rejection requires a reason
7. Demonstrate how suggested alternatives capture date and time
8. Review how the next approver or submitter is notified by workflow state changes

### Approver Questions

- Was it obvious which events need your action?
- Is the event detail page detailed enough to make a decision?
- Is anything missing that you expect to see before approving?

## HS Principal Walkthrough

### Steps

1. Log in
2. Open `Approvals`
3. Review final-step events
4. Approve one event and verify it becomes `approved`
5. Open `Calendar`
6. Confirm approved events appear there

### Questions

- Does the queue reflect the events that truly need principal action?
- Is the event information sufficient for principal review?

## Facilities Director Walkthrough

### Steps

1. Log in
2. Open `Events`
3. Review pending submissions
4. Open an event detail page
5. Demonstrate the facility conflict note workflow
6. Confirm Facilities can read but not approve

### Questions

- Is the visibility sufficient for facilities planning?
- Are conflict notes clear enough for submitters and approvers?

## Close

End each walkthrough with:

- What felt confusing?
- What slowed you down?
- What would you expect to happen next?
- Did anything feel missing?
